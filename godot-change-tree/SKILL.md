# Godot Change Tree

用可逐项核对的树说明结构范围。树只回答“哪些文件或节点以什么结构存在、这次对它采取什么动作”；行为、接口以及 Action/Event 数据流必须写在树外。

## 完成门槛

涉及文件结构、场景结构或运行时节点结构的功能设计、计划、审查和交付，必须先按本技能输出对应树，再补充树外说明。仅有行为变化且结构完全不变时，不强制输出树。

## 工作流程

1. 读取仓库、场景和相关代码，确认已有文件、保存节点、动态节点及其直接关系。
2. 区分结构变化、必要直接依赖与树外行为；不要把接口、算法或数据流塞进注释。
3. 功能设计和计划中的新增项使用已经决定的最终名称，不使用占位符；已有、修改、删除和依赖项必须以仓库事实为准。
4. 先生成树的左侧结构，再以代码块内最长左侧条目计算统一注释列。
5. 实施、审查或交付后重新读取实际结果，更新标签与说明，不能沿用失真的计划树。

## 标签

- `[创建]`：创建文件或保存到 `.tscn` 的节点。
- `[修改]`：保留原有身份并修改内容、属性、连接或生命周期。
- `[删除]`：删除已有文件或保存节点；同行说明删除原因以及原职责的去向或终止方式。
- `[动态创建]`：由代码创建或实例化、不会保存到 `.tscn` 的节点；同行说明创建者、父节点和销毁责任。
- `[依赖]`：本次不修改，但理解结构变化所必需的直接依赖文件或最小节点链；同行说明依赖原因。

标签只表示本次动作。文件移动或节点重设父级不增加“移动”标签：旧位置使用 `[删除]`，新位置使用 `[创建]`。

## 共同格式

- 每个条目必须写成 `真实名称  # [标签] 具体说明`，注释不能单独占行。
- 每个代码块以最长左侧结构条目确定注释列；树形符号、缩进、名称和节点类型全部计入左侧宽度。
- 用半角空格补齐所有条目，使每个 `#` 位于完全相同的视觉列；左侧条目与 `#` 之间至少保留两个半角空格。
- 只使用空格，禁止 Tab、独立注释行、换行续写和近似对齐。
- 不为展示完整目录或场景而列出无关文件、兄弟节点、场景分支和传递依赖。
- 不得为维持树形层级给未变化项伪造 `[修改]`；确有必要的直接依赖或最小结构链使用 `[依赖]`。

## 文件树

- 按实际父目录分块，只列变更文件和理解结构所必需的直接依赖文件。
- 叶子节点必须是实际文件名；文件名永远不用 `<>`。
- 目录只用于表达这些文件的真实父子关系，不附动作标签，也不展开无关内容。
- `[依赖]` 只允许指向直接参与挂载、实例化、继承、注册或所有权关系的文件，不列传递依赖。

## 场景树

- 仅在 `.tscn` 保存节点或运行时节点结构发生变化，或功能设计明确包含这些结构时输出。
- 第一行写实际 `res://` 场景路径，随后使用 `节点名: 节点类型` 表达保存节点。
- 只列变更节点以及表达真实父子、挂载关系所需的最小节点链；最小链中的未修改节点使用 `[依赖]`。
- 保存于 `.tscn` 的节点不用 `<>`；运行时节点始终写成 `<节点名>: 节点类型`。
- `<>` 表示节点的存在方式，标签表示本次动作，两者互不替代。停止创建原动态节点时仍保留 `<>`，并使用 `[删除]`。
- `[动态创建]` 注释必须同时交代创建者、实际父节点和销毁责任。

## 格式样例

以下名称是示例设计中已经决定的完整名称，不是占位符。真实任务必须重新核验仓库，只保留与本次结构直接相关的条目。

### 文件树

```text
res://modules/inventory/
├── inventory_system.gd          # [修改] 接管库存视图节点的创建与回收
├── inventory_model.gd           # [依赖] 由 InventorySystem 直接读取槽位状态
├── inventory_view_config.gd     # [创建] 保存库存视图场景与挂载配置
└── legacy_inventory_service.gd  # [删除] 移除旧装配入口，职责转入 InventorySystem
```

### 场景树

```text
res://modules/inventory/views/inventory_view.tscn
InventoryView: Control                  # [修改] 将槽位运行时节点统一挂载到 SlotHost
├── InventoryPanel: PanelContainer      # [依赖] 提供 SlotHost 的直接保存节点父级
│   └── SlotHost: GridContainer         # [创建] 接收 InventorySystem 创建的槽位节点
│       └── <InventorySlot>: Control    # [动态创建] 由 InventorySystem 创建并挂到 SlotHost，由 InventorySystem 在关闭页面时销毁
├── LegacySlotHost: GridContainer       # [删除] 删除旧保存节点，挂载职责转入 SlotHost
└── <LegacyInventorySlot>: Control      # [删除] 停止由 LegacyInventoryService 创建，存量实例由其退出流程销毁
```

## 最终核验

- 树只包含变更项及必要直接依赖，所有 `[依赖]` 均说明了直接原因。
- 每个代码块内所有 `#` 的列索引完全一致，且前方至少有两个半角空格。
- 文件名和保存节点未使用 `<>`；所有运行时节点都使用 `<>`，包括 `[删除]` 的原动态节点。
- `[创建]`、`[修改]`、`[删除]`、`[动态创建]`、`[依赖]` 的语义与实际状态一致。
- 文件移动或节点重设父级在旧位置标 `[删除]`、新位置标 `[创建]`。
- 结构之外的行为、接口及 Action/Event 数据流已放在树外说明。
